# issue-tracker — 이슈 통합관리

## 언제 쓰나
"반품 이슈 터졌어. 기록하고 관련된 것 묶어줘", "이거 이슈로 잡아줘", "지난 클레임과 연결해줘".
흩어진 이슈를 한 줄로 던져 관리하게 하는 기능.

## 저장 규약 (기준: `notes/README.md`)
이슈는 `notes/issues/` 에 저장한다 — `quick-note`·`recall` 의 분류 규약과 동일한 폴더다.
경로 표기: `notes/issues/YYYY-MM-DD-짧은제목.md` (예: `notes/issues/2026-07-20-반품-클레임.md`)

## 동작 순서
1. **이슈 파악** — 무슨 일이, 누구와, 얼마나 급한지. **최소 정보(무슨 일 + 누구/어디)가 없으면**
   CLAUDE.md 공통 원칙대로 파일을 만들지 말고 먼저 되묻는다(빈 이슈는 추적을 오염시킨다).
2. **기록** — `notes/issues/` 에 아래 형식으로 저장. 같은 이슈가 이어지면 기존 파일에 이어 적는다.
3. **관련 묶기 — 양방향(의무)** — 같은 고객·주제의 기존 기록을 `notes/` 전체에서 찾아(=recall 규약)
   **연결**한다: ① 새 이슈 파일의 "관련" 항목에 기존 파일 경로를 적고, ② **연결한 기존 파일에도
   "관련: <새 이슈 경로>" 한 줄을 추가한다(역링크).** 어느 쪽에서 열어도 서로를 찾을 수 있어야
   "묶었다"고 말할 수 있다. "묶어줘"의 의미: 같은 사안이 이어지는 것이면 기존 이슈 파일에 이어 적고,
   새 사안이면 새 파일 + 양방향 연결.
4. **상태·다음 액션 제안** — 지금 상태(열림/진행/해결)와 대표가 할 다음 한 가지를 제안한다.
5. **열린 항목 등록(todos)** — 새 이슈를 개설할 때 `state/status.json` 의 `todos[]` 에 **`open`** 으로
   추가한다: `{ "kind":"issue", "title":<이슈 제목>, "due":null, "source":<이슈 파일 경로>,
   "state":"open" }`. (배열이 없으면 새로 만든다.) `daily-briefing` 이 이 배열을 읽어 미해결 이슈를
   표면화한다.
   - **해결 처리:** 이슈가 해결되면(상태를 "해결"로 바꿀 때) 같은 항목의 `state` 를 **`done`** 으로
     바꾼다(지우지 않는다). 대상은 `source` 경로로 특정한다.
6. **상태 갱신** — `state/status.json` 의 `today.counts.issues` +1, `today.activities` 에
   `{time,type:"issue",summary}` 추가, `updatedAt`·`lastActiveAt` 갱신 후
   **`./refresh-dashboard.sh` 실행**으로 `state/dashboard-data.js` 재생성 (`docs/STATE_CONTRACT.md` 준수).

## 저장 파일 형식
```markdown
# 반품 클레임 — A고객

- 날짜: 2026-07-20 16:00
- 분류: issues
- 상태: 열림
- 관련: notes/customers/2026-07-12-A고객-주문.md

CS에서 반품 요청 접수. 사유: 사이즈 불일치.
다음 액션: 교환/환불 방침 확인 후 A고객에게 회신.
```

## 응답 예시 (텔레그램)
```
이슈 잡았어요 → notes/issues/2026-07-20-반품-클레임.md (상태: 열림)
연결: 7/12 A고객 주문 기록과 묶었어요.
다음 할 일: 교환/환불 방침 정하고 오늘 중 회신 → 정해지면 알려주세요, 초안 잡아드릴게요.
```
