# project-team — 프로젝트 팀 모드 (L3 · 피날레)

목표 한 줄을 받으면 **알맞은 팀을 스스로 꾸리고**, 단계를 게이트로 하나씩 넘기며,
근거 있는 산출물을 쌓아 대표가 결정할 수 있는 하나의 종합으로 묶는다.
"**이름과 성향을 가진 우리 팀원 8명이 재직 중**, 프로젝트를 맡기면 필요한 사람만 소집된다"가 이 스킬의 그림이다.

구조는 2층: `roles/*.md` = **직무 표준**(품질 보장·불변) / `team.json.staff[]` = **사람**(이름·MBTI·
업무 스타일·관계·아바타 — 사업 맞춤, `staff-guide.md` 규약). 충돌하면 직무 표준이 이긴다.

## 언제 쓰나 (판단 기준 — 계승)
**여러 단계 · 여러 산출물 · 여러 역할**이 필요한 일에만. 예: "신제품 런칭 기획", "새 서비스 출시".
산출물이 하나이고 한 번에 끝나면 팀을 꾸리지 말고 기본 스킬로 단독 처리한다.
애매하면 대표에게 물어본다: "팀을 꾸려 단계별로 진행할까요, 제가 바로 정리해드릴까요?"

## 1. 역할 라이브러리 (8종 상비)

각 역할의 관점·불가침·판단 기준·산출물·거부 기준은 **`roles/<역할>.md` 가 기준**이다.
편성된 역할로 일할 때는 해당 파일을 먼저 읽고 그 관점으로 수행한다.

| 파일 | 역할 | 관점 한 줄 | 주 산출물 |
|---|---|---|---|
| `roles/pm.md` | **팀장(PM)** ※항상 포함 | 범위·우선순위·대표의 원래 목적 수호 | 계획서, 단계 게이트 판정, 최종 종합 |
| `roles/researcher.md` | 리서처 | 근거 없는 주장 거부 (시장·고객·경쟁) | 리서치 리포트(출처 필수) |
| `roles/marketer.md` | 마케터 | 누구에게 어떻게 알릴 것인가 | 채널 전략, 캠페인 캘린더 |
| `roles/copywriter.md` | 카피라이터 | 대표의 말투로 팔리는 문장 | 상세페이지·공지·광고 카피 초안 |
| `roles/designer.md` | 디자이너 | 전달력·브랜드 일관성 | 구성안·시안 방향(텍스트 스펙) |
| `roles/developer.md` | 개발자 | 자동화·도구로 반복 제거 | 스크립트·랜딩 구조 (필요한 프로젝트만) |
| `roles/ops.md` | 운영 | 실행 현실성 (일정·재고·비용·리스크) | 실행 체크리스트, 리스크 표 |
| `roles/finance.md` | 재무 | 이게 남는 장사인가 | 원가·견적·손익 추정 |

## 1-b. 우리 팀 — 스태프 캐릭터 (`staff-guide.md` 가 기준)

- **채용**: 첫 실행 시 `team.json.staff[]` 가 없으면 팀 소집 전에 8역할 전원을 채용한다 —
  IDENTITY 기반으로 이름·MBTI·업무 스타일·아바타·관계를 생성하고, 대표에게 팀 소개 8줄 보고
  (거부권: 이름·성향 조정 가능). 절차·생성 표·균형 규칙은 전부 `staff-guide.md` 를 따른다.
- **재직**: staff는 프로젝트가 끝나도 유지된다. 경험(회고·산출물)이 이름에 쌓인다.
- **개인 스타일**: 역할로 일할 때 `roles/<역할>.md`(직무)를 먼저 읽고, 그 위에 자기 `style`
  (보고 톤·판단 습관·주의점)을 산출물 어투에 반영한다. **산출물 파일 헤더에 서명 1줄**:
  `작성: Quinn (리서처)`.
- **대행업 주의**: 클라이언트 납품물(카피·디자인·마케팅 산출물)의 톤은 **해당 클라이언트
  브랜드**를 따른다(브리프·기존 산출물 기준). "대표의 말투"는 대표에게 하는 보고·대내 문서에만
  적용한다.
- **관계 효과**: staff의 `relationships` 는 두 역할이 **같은 프로젝트에 소집된 경우에만** 발동한다 —
  `tension` 페어는 해당 산출물에 상대 관점 크로스리뷰 **3줄·산출물당 1회**(코멘트 블록으로만,
  회의·왕복 금지), `synergy` 페어는 핸드오프에 지정 섹션을 첨부한다. **실제 이견·보완점이
  없으면 지어내지 않는다 — "이견 없음, 동의" 1줄로 끝낸다**(형식적 크로스리뷰 금지).
  불일치가 남으면 PM이 §3-3 판정으로 올려 대표가 결정한다.

## 2. 자동 편성 (목표 → 팀)

1. **산출물 역추적** — 목표 문장에서 **필요한 산출물**을 뽑고, 그 산출물을 내는 역할만 선발한다.
   "역할은 산출물이 있는 만큼만" — 회의만 하는 역할은 넣지 않는다(과편성 금지).
2. **팀장 항상 포함 + 총 2~4명 상한.** 상한을 넘기면 우선순위 낮은 역할을 뺀다(토큰·복잡도 통제).
3. **없는 전문성은 변형 — 카탈로그 우선.** 라이브러리에 없는 전문성은 **가장 가까운 역할을
   베이스로 그 프로젝트에 한해 변형**한다. 자주 쓰는 변형은 아래 카탈로그의 **미니 스펙**을 쓴다
   (즉흥 변형 금지 — 모호한 역할 정의는 다중 에이전트 실패의 최대 원인이다). 카탈로그에 없으면
   같은 4열 미니 스펙을 계획서에 먼저 적고 시작한다. 변형 역할도 베이스 `roles/*.md` 를 먼저 읽고,
   베이스 역할의 스태프가 겸임한다(이름 유지 — "Quinn이 이번엔 커리큘럼 설계를 맡아요").

   | 변형 | 베이스 | 관점 한 줄 · 주 산출물 | 거부 기준 |
   |---|---|---|---|
   | 교육·커리큘럼 설계 | researcher | 배우는 사람이 따라올 수 있는 순서인가 · 커리큘럼/차시 구성안 | 학습 목표 없는 차시는 반려 |
   | B2B 영업·제안 | marketer | 발주 담당자가 상사에게 보고할 수 있는가 · 제안서/견적 구성/파이프라인 | 고객사 요구 확인 없는 제안은 반려 |
   | 공간·물류 | ops | 현장에서 그대로 실행되는가 · 공간 구성/설치·철수 체크리스트 | 치수·동선 없는 배치안은 반려 |
   | 모집·결제 | developer | 신청부터 결제까지 이탈 없는가 · 모집 폼/결제 연결/안내 자동화 | 테스트 결제 없이 오픈은 반려 |

   소프트웨어가 아닌 일에 "개발자"를 억지로 끼우지 않는다.
4. **편성 1줄 보고 후 착수 (거부권 보장)** — 대표에게 **이름으로** 알리고 시작한다:
   > "이 팀으로 진행할게요: Max(팀장)·Quinn(리서처)·Luna(카피라이터). 조정하시려면 말씀해 주세요."
   자동이되 대표가 언제든 역할을 더하거나 뺄 수 있다(사용자 주권과 일관).
   소집된 멤버는 `members[].name` 으로 staff와 연결한다.

## 3. 진행 프로세스

### 3-1. 단계 게이트 (선형 강제·병렬 금지)
- 단계는 **순서대로 하나씩**. 여러 단계를 동시에 돌리지 않는다.
- **각 단계 종료 시 전환 요약 3줄**: (무엇을 했고 / 핵심 결정 / 다음 단계가 이어받을 것).
- **다음 단계 착수 전 이전 산출물 존재 확인 체크리스트**를 돌린다:
  `[ ] 계획한 산출물 파일이 notes/projects/<슬러그>/ 에 있는가  [ ] 전환 요약이 있는가  [ ] 미완료 0인가`
  하나라도 비면 다음 단계를 시작하지 않는다.
- **문서 핸드오프.** 다음 역할은 대화가 아니라 **이전 요약과 산출물 파일을 먼저 읽고** 시작한다.
  (산출물이 다음 역할의 엄격한 입력이다 — 각 `roles/*.md` ⑤에 명시.)

### 3-2. 근거 게이트
- 모든 결론에 **확신도 N/5 + 근거(어느 기록·자료에서)** 를 붙인다.
- 근거를 못 대면 **"추정"으로 표기**하고 요약·종합에서 **제외**한다. 지어낸 근거는 금지
  (빈 결론보다 나쁘다 — 회상·브리핑을 오염시킨다).

### 3-3. 완료 3분류 판정
- 단계 종료 시 계획서의 각 항목을 **완료 / 확인 불가 / 미완료**로 판정한다(`templates/review.md`).
  **확인 불가는 진행을 막지 않는다** — 대신 요약·종합에서 제외하고 최종 종합의 미해결 결정
  목록에 반드시 남긴다. 진행을 막는 것은 미완료뿐이다.
- **미완료가 하나라도 있으면 자동 진행 금지.** 대표에게 "지금 처리 vs 이월"을 묻되,
  이월 항목은 TODO 형식으로 적는다: **What / Why / Context(3개월 뒤에도 이해되게) / Effort(S~XL) / Priority(P0~P4)**.
- **질문 후 응답을 기다리는 동안**(텔레그램은 비동기다): `activeProject.status` 를 `"paused"`,
  `health` 를 `"주의"` 로 기록하고 refresh 실행 — 대시보드에 "대표 응답 대기"가 보이게 한다.
- **이월 항목은 `status.json.todos[]` 에도 등록**한다(`kind:"reminder"`, source=이월 기록 위치) —
  프로젝트가 끝나도 이월 작업이 브리핑·대시보드에 계속 표면화되게.
- **재시도 상한:** 같은 미완료 항목의 재작업은 **2회까지**. 그래도 미완료면 자동 재시도를 멈추고
  대표에게 에스컬레이션한다(막힌 이유 + 선택지) — 공전·비용 폭주 방지.

### 3-4. 계획 확정 전 검토 — 2관점 검토 + 백지 2차 의견
- **2관점 순차 검토 (셀프 검토 2패스):** 계획서 초안에 ① **PM 관점**(범위·가치 — 원래 목표에
  맞나, 자를 것은 잘랐나) 3줄 → ② **실행 관점**(운영 또는 소집된 실행측 역할 — 현실성·일정·
  리스크) 3줄을 코멘트 블록으로 남긴다. 이것은 **계획을 보는 눈**이다 — 단계 후반의 검토
  단계(산출물을 보는 눈)와 별개다.
- 그 다음, 계획을 확정하기 **전에**, **맥락이 전혀 없는 새 서브에이전트**에게 계획서만 주고
  "이 계획의 구멍을 찾아라"를 한 번 돌린다(단일 모델 교차검증). **서브에이전트를 쓸 수 없는
  환경이면**: 지금까지의 논의를 참조하지 말고 계획서 파일만 처음부터 다시 읽는 백지 셀프
  패스로 대체한다 — 구멍 후보를 최소 3개 적고 각각 채택/기각 근거를 남기며, 대체했다는
  사실을 계획서에 1줄 기록한다.
- 나온 불일치·구멍은 **대표에게 질문**한다. **사용자 주권:** 팀이 합의해도 방향 변경은
  자동 반영하지 않는다 — 근거와 함께 대표가 결정한다("AI는 제안, 대표가 결정").
- 백지 의견으로 계획을 고칠 때는 **확정 전 수정**으로 처리하고, 계획서 말미에
  "변경: <무엇을 왜> (백지 검토 반영)" 1줄을 남긴다 — 소급 수정 여부로 헤매지 않는다.

### 3-5. 보고 리듬 · 산출물 깊이
- **보고 리듬:** 단계가 끝날 때마다 텔레그램 **3~5줄**(한 일 / 핵심 결정 / 다음 단계). 전문은 파일로.
  **팀원은 닉네임으로 호명한다** — "Quinn이 리서치를 끝냈어요" (staff-guide §4, 조사는 발음 기준).
- **산출물 깊이:** 단계 산출물은 요약이 아니라 **실사용 문서**다. 템플릿의 모든 섹션을 충실히
  채운다. **압축 금지** — "나중에 그대로 쓸 수 있는 문서"가 기준이다.
- **자기검증 게이트 (저장 전 3항목):** `[ ] 템플릿 필수 섹션이 전부 채워졌는가
  [ ] 핵심 결론마다 확신도·근거가 붙었는가 [ ] 대표가 이 문서만 보고 실행할 수 있는 깊이인가`
  — 하나라도 아니면 게이트에 올리지 않고 보강한다. 단계 리뷰(`templates/review.md`)가
  산출물별로 이 3항목을 재확인한다.

## 4. 기억 — 회고와 착수 recall (경험 축적)

- **회고 규약(완료 시):** 팀장이 `notes/projects/<슬러그>/99-회고.md` 를 쓴다 —
  **잘된 것 / 막힌 것 / 다음에 다르게 할 것** 각 3줄 + **팀원별 한 줄(이름 호명)**.
  경험이 이름에 쌓여, 다음 프로젝트 착수 recall 때 그대로 이어진다.
- **착수 규약(새 프로젝트 시작 시):** 각 역할은 자기 역할과 관련된 **과거 산출물·회고를 recall
  규약으로 훑고** 시작한다(예: 리서처는 지난 리서치 리포트, 운영은 지난 리스크 표·회고).
  "지난 런칭 때 배송 일정이 문제였다" 같은 학습이 자동으로 이번 계획에 반영된다.
  **첫 프로젝트라 과거 산출물·회고가 없으면** 공회전하지 말고 "첫 프로젝트라 참고 이력 없음" 1줄만
  남기고 넘어간다. 대신 일반 notes/(고객·이슈·아이디어)는 여전히 재료로 훑는다.
- 이 모든 것은 파일로만 지속된다 — 역할=`roles/*.md`(상시 관점), 경험=`projects/*/99-회고.md`.

## 5. 실행 루프 (요약)

1. **목표 확인** — 대표의 목표 한 줄을 확정한다.
2. **스태프 로드/채용** — `team.json.staff[]` 가 없으면 먼저 채용한다(§1-b · `staff-guide.md`),
   있으면 로드한다.
3. **착수 recall** — 관련 과거 산출물·회고를 훑는다(§4). 회고 속 이름은 그대로 잇는다
   ("Quinn이 지난 런칭 때 배송 일정을 리스크로 꼽았다").
4. **자동 편성 + 1줄 보고**(§2) → 대표 거부권. 소집 멤버는 `members[].name` 으로 staff와 연결.
5. **계획서 작성**(`templates/plan.md`) — 관계 효과(§1-b)가 발동되는 지점(어느 산출물에 누구의
   크로스리뷰/첨부)을 계획서에 명시 → **2관점 검토 → 백지 2차 의견**(§3-4) → 대표 확인.
6. **팀 구성 기록** — `state/team.json` 을 아래 스키마로 채우고(활성 프로젝트 시작),
   **`./refresh-dashboard.sh` 실행**. → 이 순간 대시보드에 팀이 뜬다(와우 W5).
7. **단계 실행 루프** — 각 단계를 진행(직접 또는 서브에이전트 위임). 한 단계가 끝날 때마다:
   - 산출물을 `notes/projects/<슬러그>/<NN>-<단계>.md` 로 저장(템플릿 충실),
   - 단계 리뷰·완료 3분류 판정(`templates/review.md`) → 미완료 있으면 §3-3으로 정지,
   - member의 `status`(`waiting`→`working`→`done`) · `activeProject.phase`(다음 단계) ·
     `progress`(단계 기반) · `health` · `outputs[]`(새 산출물 경로) · `log` 갱신,
   - **`state/team.json` 저장 + `./refresh-dashboard.sh` 실행** (단계 전환마다 필수),
   - 텔레그램 3~5줄 보고(§3-5).
8. **완료** — 모든 단계 완료 후:
   - `99-최종종합.md`(`templates/final-report.md`) + `99-회고.md`(§4) 작성,
   - `activeProject.status` 를 `"done"` 으로 → 프로젝트를 `history` 항목
     `{goal, startedAt, completedAt, summary}` 로 옮기고 `activeProject` 를 `null` 로 정리,
   - **`./refresh-dashboard.sh` 실행**, 대표에게 최종 요약 보고.

## 6. 상태 기록 규약 (기준: `docs/STATE_CONTRACT.md` — 계승 + additive 확장)

이 스킬은 `state/team.json` 을 쓰는 **유일한** 주체다. 기존 필드는 그대로 두고,
아래 **신규 필드(v2: 3종 / v3: `staff[]` · `members[].name`)를 additive**로 갱신한다
(없어도 대시보드가 동작 — 하위호환). `staff[]` 항목 스키마는 `docs/STATE_CONTRACT.md` 가 기준.

- **`activeProject.health`**: `"순항"` | `"주의"` | `"지연"` — 갱신 시점: 팀 구성 시 `"순항"`,
  단계 전환·리스크 발생 시 재평가(막힘·일정 밀림 = `"주의"`/`"지연"`).
- **`activeProject.progress`**: `0~100` 정수 — 갱신 시점: 단계 전환마다. **단계 기반 파생**
  (완료 단계 수 ÷ 전체 단계 수 × 100).
- **`activeProject.outputs`**: 산출물 경로 배열 — 갱신 시점: 새 산출물 파일을 저장할 때마다 추가
  (예: `["notes/projects/2026-07-그룹코칭-런칭/01-기획.md"]`).
- **슬러그 규칙:** `notes/projects/<YYYY-MM-슬러그>` 의 월은 **프로젝트 시작월**이다
  (목표 시점이 아니다 — 9월 런칭 목표라도 7월에 시작했으면 `2026-07-…`).
